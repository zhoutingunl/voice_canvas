## 1. 项目简介与设计目标

VoiceCanvas 是一款 **Voice First** 的绘图工具。用户全程不碰鼠标和键盘，仅靠语音完成图形的**创建、编辑、布局、生成与导出**。

试题强调四个考点，本设计逐一对应：

| 考点 | 本设计的应对 |
| --- | --- |
| 指令理解的**准确性与容错性** | 三层容错（本地归一化 + LLM 容错解析 + 低置信度澄清回问），见 §10 |
| 语音到绘图的**响应延迟** | 规则快路径 + 乐观渲染 + 占位骨架 + 异步回填，分层延迟指标，见 §9 |
| **复杂指令的拆解与执行** | LLM 把一句话拆成原子指令数组；语义场景再做"场景图分解"，见 §7/§8 |
| 工程完整性 | 可插拔架构、密钥不入库、PR 协作、QoS/评估指标，见 §5/§14/§16/§19 |

---

## 2. 设计原则

1. **Voice First**：所有绘图创作动作必须可被语音完成；鼠标/键盘仅用于"创作流程之外"的开发者配置页（见 §10、§11 的边界说明）。
2. **容错优先**：宁可澄清，不可瞎画。ASR 必然出错，系统要"知道自己不确定"。
3. **低延迟体感**：能本地做的不上云，能先画的先画，慢操作异步回填。
4. **可插拔**：ASR / LLM / 生图都抽象成接口，可在免费与付费供应商之间切换。
5. **密钥不入库**：任何 API Key 绝不进入 Git 仓库（硬性要求）。

## 3. 指令能力规划（三档）

能力分三档。第 1、2 档计划实现，第 3 档明确**不做**并说明原因。

### 第 1 档 · 核心能力（计划全做）

| 能力 | 指令示例 |
| --- | --- |
| 创建图形：圆 / 矩形 / 椭圆 / 直线 / 三角形 / 文字 | "画一个蓝色的圆" |
| 颜色（填充 / 描边） | "画个红色方块"、"把边框改成绿色" |
| 尺寸（大 / 小 / 具体数值） | "画一个大圆"、"半径 50 的圆" |
| 位置（九宫格 + 相对移动） | "在右上角画一个三角形"、"往右移一点" |
| 删除 / 清空画布 | "删掉那个圆"、"清空画布" |
| 撤销 / 重做 | "撤销"、"恢复" |

### 第 2 档 · 进阶能力（计划全做，重点亮点）

| 能力 | 指令示例 | 对应考点 |
| --- | --- | --- |
| **批量 + 布局**（一句拆多步） | "画三个圆排成一行"、"画一个 2×2 的方格" | 复杂指令拆解 |
| **相对引用 / 上下文指代** | "把刚才那个往右移"、"选中红色的圆" | 准确性 + 上下文理解 |
| 修改已有图形（移动 / 缩放 / 改色 / 旋转） | "把那个方块放大一点"、"旋转 45 度" | 执行能力 |
| **语义场景生成（场景图分解）** | "画一只躺着的加菲猫，靠在黑色桌子边，桌上有一盆花" → "花改成盛开的牡丹" | 复杂拆解 + 多轮上下文 + 增量执行 |

### 第 3 档 · 明确不做（写明原因）

| 能力 | 不做的原因 |
| --- | --- |
| 自由手绘曲线 / 任意路径 | 纯语音难以精确描述轨迹坐标，投入产出比低 |
| 纯文生图（整句直接出一张图，不做拆解） | 会把"拆解 / 延迟 / 容错"全部外包给生图模型黑盒，**反而掩盖本题要考核的工程能力**，是减分项 |
| 图层管理 / 分组 / 多格式批量导出 | demo 范围外，时间有限 |
| 多人协同编辑 | 超出单机 demo 范畴 |

## 4. 指令 DSL 设计

ASR 给出一句中文文本，LLM 把它解析成**指令数组**，每条是一个原子操作。第 1 档（向量图形）与第 2 档语义实体**共用同一套指令信封**，仅靠 `kind` 字段分流：

```jsonc
{
  "op": "create",            // create | update | delete | move | transform | clear | undo | redo —— 共用
  "id": "cat1",              // 目标 / 新建对象 id
  "kind": "shape",           // shape（向量原子，同步渲染） | entity（语义实体，异步生图）

  // kind = shape 时：
  "shape": "circle",          // circle | rect | ellipse | line | triangle | text
  "geo":  { "r": 40 },        // 几何参数：{r} / {w,h} / {x2,y2} ...
  "fill": "red", "stroke": "green",

  // kind = entity 时：
  "entity": "加菲猫",
  "prompt": "一只躺着的加菲猫，卡通风，透明背景",   // 交给文生图模型
  "pose":   "躺",

  // 通用字段：
  "pos": { "x": 150, "y": 200 },   // 绝对坐标，或九宫格枚举 "top-right" 等
  "rel": "在 table 上",            // 可选：相对关系，需经"关系→坐标 resolver"求解（见 §8）
  "z":   2,                         // 层级
  "confidence": 0.92                // 解析置信度，低于阈值触发澄清（见 §10）
}
```

**示例**：「画三个红色的圆排成一行」解析为：

```jsonc
[
  {"op":"create","kind":"shape","shape":"circle","fill":"red","geo":{"r":40},"pos":{"x":150,"y":200},"id":"c1"},
  {"op":"create","kind":"shape","shape":"circle","fill":"red","geo":{"r":40},"pos":{"x":260,"y":200},"id":"c2"},
  {"op":"create","kind":"shape","shape":"circle","fill":"red","geo":{"r":40},"pos":{"x":370,"y":200},"id":"c3"}
]
```

> **关键收益**：`op` 信封、历史栈、撤销/重做、指令队列对 `shape` 与 `entity` 完全共用；二者只在"渲染方式"和"同步/异步"上不同。撤销/重做即"重放/回退指令"，与 DSL 天然契合。

## 5. 系统架构

```text
┌─────────┐   语音    ┌──────────────────────┐
│ 麦克风   │ ───────▶ │ ASR 层 IASREngine    │  Web Speech API（免费/桌面）
└─────────┘           │  · WebSpeechASR      │  阿里百炼 ASR（云端/手机/付费）
                      │  · BailianASR        │
                      └──────────┬───────────┘
                          文本流（partial/final）
                                 ▼
                      ┌──────────────────────┐
                      │ L1 归一化（本地）      │  繁简/数字归一 · 去语气词 · 同音词典
                      └──────────┬───────────┘
                                 ▼
                      ┌──────────────────────┐
            命中 ◀──── │ 规则快路径（本地）     │  高频简单句 → 直接出指令（<50ms）
                      └──────────┬───────────┘
                          未命中/复杂句
                                 ▼
                      ┌──────────────────────┐
                      │ L2 语义解析 MiniMax-M3 │  自然语言 → 指令 JSON（含场景图分解）
                      └──────────┬───────────┘
                          低置信度 → L3 澄清回问
                                 ▼
                      ┌──────────────────────┐
                      │ 执行引擎（乐观渲染）    │  指令队列 + 历史栈 + 关系 resolver
                      └─────┬──────────┬──────┘
                  kind=shape│          │kind=entity（异步）
                            ▼          ▼
                   ┌─────────────┐  ┌──────────────────────┐
                   │ Konva 画布   │◀─│ MiniMax image-01-live │  生图 → 抠图 → 合成
                   │ (react-konva)│  │ + 预置素材库（兜底）   │
                   └─────────────┘  └──────────────────────┘

后端（Flask 薄代理，持有 Key）：/api/asr  ·  /api/parse  ·  /api/imagine
```

前端（浏览器 / WebView）只调用自己的后端代理，**不接触任何第三方 Key**。

## 6. ASR 层（语音转文字）

抽象统一接口 `IASREngine`，对外吐出 `onPartial(text)` / `onFinal(text)` 文本流，上层不关心底层实现。

| 运行环境 | 默认 ASR | 说明 |
| --- | --- | --- |
| 桌面浏览器（Chrome / Edge） | **Web Speech API**（免费） | `lang='zh-CN'`、`interimResults` 流式中间结果，体感快 |
| 手机 App（WebView 套壳） | **阿里百炼 ASR**（云端，**必须**） | WebView 不提供原生 `SpeechRecognition`，免费方案在此不可用 |
| 不支持 Web Speech 的浏览器 | 自动 fallback 到百炼 | 容错兜底 |
| "付费用户"演示 | 手动切换到百炼 | 模拟产品付费路径，精度更高 |

> **已知约束**：Web Speech API 依赖云端引擎、浏览器兼容性有限（Firefox/Safari 残缺）、且**在 WebView 内不可用**——这正是"手机端强制走百炼"的根因。

> **选型说明**：MiniMax 平台提供 LLM 与文生图，但**不提供语音转文字（ASR）**，因此 ASR 这一入口环节由 Web Speech / 阿里百炼承担。

## 7. 指令理解层（Hybrid 解析）

采用 **本地规则快路径 + LLM 慢路径** 的混合方案，兼顾延迟与容错：

1. **规则快路径（本地，<50ms）**：正则/关键词命中高频简单句（"画圆"、"清空"、"撤销"），直接产出指令，零网络往返。
2. **LLM 慢路径（MiniMax-M3）**：规则未命中或句子复杂时，调用 MiniMax-M3 把自然语言转成 §4 的指令 JSON。M3 原生多模态、1M 上下文，足以承载画布上下文与多轮对话。
3. **容错 Prompt**：在系统提示中明确告知"用户在语音绘图，输入可能存在识别错误，请结合上下文猜测真实意图，并输出标准化指令；不确定时降低 confidence"。

> 与 ASR 同思路，LLM 也抽象为 `ILLMParser` 接口，便于在 MiniMax 与其它供应商间切换。

## 8. 复杂指令拆解（场景图分解）

这是第 2 档的核心亮点，对应"复杂指令的拆解与执行"考点。

以「画一只躺着的加菲猫，靠在黑色桌子边，桌上有一盆花」为例，LLM 先拆成**结构化场景图**（实体 + 属性 + 空间关系 + 层级）：

```jsonc
[
  {"id":"table",  "kind":"entity","entity":"桌子",   "color":"黑", "pos":"bottom-center", "z":1},
  {"id":"cat",    "kind":"entity","entity":"加菲猫", "pose":"躺",  "rel":"靠在 table 左边", "z":2},
  {"id":"pot",    "kind":"entity","entity":"花盆",   "rel":"在 table 上",                "z":2},
  {"id":"flower", "kind":"entity","entity":"花",     "rel":"在 pot 里",                  "z":3}
]
```

执行步骤：

1. **关系 → 坐标 resolver**：把 `rel`（"靠在 table 左边"、"在 pot 里"）求解为绝对坐标与层级，确定每个 sprite 的摆放。
2. **逐实体渲染**：每个实体用 MiniMax `image-01-live` 生成卡通 sprite，按坐标和 `z` 合成到画布。
3. **增量修改**：后续「花改成盛开的牡丹」→ 只更新 `flower` 这一个实体并重渲染，其余不动。

> 这一设计同时展示了**拆解（一句话→场景图）+ 上下文理解（多轮指代）+ 增量执行（只重渲染改动实体）**三种能力，而**拆解、布局、增量更新都由本系统完成**，生图模型仅充当"渲染笔刷"。

## 9. 渲染与执行引擎（延迟模型）

执行引擎采用**乐观渲染 + 占位骨架 + 异步回填**，把"生图慢"的劣势转化为"拆解可视化 + 秒级反馈"的体验亮点：

1. **乐观渲染**：LLM 返回指令数组后，立即画出所有能同步完成的部分（`kind=shape`、布局框架、`entity` 的占位骨架），用户秒见反馈。
2. **占位骨架**：`entity` 生图期间，在目标位置画一个带标签的灰色占位框（如「🐱 加菲猫 · 生成中…」），既给即时反馈，又把拆解结果可视化。
3. **异步回填**：`image-01-live` 的真图 2–5s 后台返回，替换占位框。
4. **增量更新**：修改单个实体只重渲染该 sprite。

**分层延迟指标（量化"响应延迟"考点）**：

| 路径 | 目标延迟 |
| --- | --- |
| 规则快路径（简单图形） | < 50 ms |
| LLM 解析 + 同步图形落地 | ~ 0.5–1 s |
| 实体生图（占位即时，真图异步回填） | 占位 < 100 ms，真图 2–5 s |

**画布引擎**：基于 **Konva.js（react-konva）**——对象模型画布，每个图形带 id、可选中/移动/缩放/层级，`Konva.Image` 直接承载 entity sprite，与指令 DSL 的"按 id 操作对象"天然对齐。

> **生图风险与兜底**：合成需要**透明背景** sprite，`image-01-live` 是否原生支持透明背景需实测；若不支持，兜底为出图后前景抠图（去白底），或退回**预置素材库**（加菲猫/牡丹/桌子等 SVG/PNG）。生图主路径走云端，**网络/额度/抠图异常时自动 fallback 到素材库**，确保 demo 不翻车。

## 10. 容错与澄清机制

ASR 文本不可靠，是纯语音工具最大的敌人。三类错误 + 三层防御：

**错误类型**：① 同音/近音错字（"圆"→"元/园"、"矩形"→"桔形"）；② 口语/多样表达（"来个圈"、"整一个球"=画圆）；③ 半句/犹豫/噪声（"呃……画那个……圆吧"）。

| 层 | 做法 | 兜住什么 |
| --- | --- | --- |
| **L1 归一化**（本地，零延迟） | 去语气词、繁简/数字归一、**自定义同音词典**（元/园→圆，桔形→矩形） | 高频同音错字、口语杂音 |
| **L2 LLM 容错解析**（MiniMax-M3） | Prompt 声明"输入可能有识别错误，猜测真实意图并输出标准指令" | 多样表达、复杂/模糊句、上下文指代 |
| **L3 澄清回问**（简版兜底） | 置信度低于阈值时**不硬画**，反问一句"你是想画圆吗？" | 防止瞎执行，把不确定暴露给用户 |

**同音词典** 采用**外置 JSON + 配套可视化配置页**，可随时加词/改词，便于现场救场。

> **边界说明（自圆其说）**：配置页使用鼠标键盘，看似与"绘画创作不能用鼠标键盘"冲突。本约束针对的是**绘图创作流程**；配置页属于**开发者/管理员的设置工具，处于创作流程之外**，不违背 Voice First 原则。

## 11. 交互形态：无鼠标键盘下的麦克风触发

不能"按住说话"（需鼠标），因此采用**常驻聆听**：

- **常开 + VAD 自动断句**：麦克风全程开启，靠语音活动检测（VAD）自动切句，每个完整句作为一条指令。
- **醒目的聆听态动效**：大号、触屏友好的"聆听中"指示器，状态清晰。
- **语音开关**：说"停止聆听"/"开始聆听"控制麦克风，依旧零鼠标键盘，demo 时避免误接环境噪声。
- **唤醒词（可选增强）**：可选开启"小画小画…"唤醒门控以抗噪（Web Speech 无原生唤醒词，由本地匹配实现）。

## 12. 前端适配与 App 套壳

前端从一开始即按**响应式**设计，做到"几乎零工作量套壳"为手机 App：

- **响应式布局**：画布随视口自适应，`<meta viewport>`，指示器大号触屏友好。
- **Konva 原生支持移动端**（高 DPI、触摸事件、响应式 resize）。
- **WebView 套壳**：Android WebView / iOS WKWebView 直接加载网页，业务代码零改动。
- ⚠️ **唯一非零工作量点 —— WebView 麦克风权限**：WebView 不自动授予网页麦克风权限，原生壳需手动放行（Android 实现 `onPermissionRequest`；iOS 配 `NSMicrophoneUsageDescription` 并允许媒体捕获）。几行原生代码。
- **PWA（可选）**：加 manifest + service worker 即可"添加到主屏幕"当 App 用，作为免套壳的备选路径。

## 13. 技术栈总览

| 层 | 选型 | 理由 |
| --- | --- | --- |
| 前端框架 | React + Vite + TypeScript | 组件化适合"画布 + 聆听指示器 + 配置页"，Vite 构建快 |
| 画布库 | **react-konva（Konva.js）** | 对象模型画布，与"按 id 操作图形"的 DSL 天然对齐 |
| ASR | Web Speech API（免费）+ 阿里百炼（云端/手机/付费） | 抽象 `IASREngine`，可插拔 |
| 指令解析 | **MiniMax-M3** | 原生多模态、1M 上下文；与生图同一供应商 |
| 实体生图 | **MiniMax image-01-live** | 支持卡通/手绘风，卡通 sprite 便于抠图合成 |
| 语音回执（可选） | MiniMax Speech-2.8（TTS） | 朗读澄清/回执，锦上添花 |
| 后端 | **Python Flask**（极薄代理） | `/api/asr`、`/api/parse`、`/api/imagine` 三端点转发，持有 Key；`python-dotenv` 读 Key，`flask-cors` 处理跨域 |
| 状态/历史 | 自管指令历史栈 | 撤销/重做 = 指令重放，契合 DSL |

## 14. 安全与密钥管理

**硬性要求：任何 API Key 绝不进入 Git 仓库 `git@github.com:zhoutingunl/voice_canvas.git`。**

- 所有 Key（MiniMax、阿里百炼）只存于后端 `.env`，并写入 `.gitignore`；仓库仅保留 `.env.example`（占位、无真值）。
- 前端纯浏览器运行，**不接触任何 Key**；一切第三方调用经由 Flask 薄代理转发。这同时解决了跨域与"手机 WebView 调云端"问题。
- 禁止提交：`API_KEY`、`AccessToken`、`Cookie` 及任何认证信息。

## 16. 质量指标：QoS / 评估 / 埋点

**QoS 目标**：首响应 < 1s；简单图形 < 500ms；复杂场景占位 < 100ms、真图 < 5s；ASR 成功率 > 90%；命令执行成功率 > 90%。

**评估指标（避免伪指标）**：
- Command Accuracy（命令理解正确率）
- Drawing Accuracy（图形结果符合预期比例）
- Task Completion（任务完成率）
- Clarification Rate（澄清触发比例）
- Undo Rate（撤销比例，越低越好——反映理解质量）

**埋点事件**：`voice_start/stop`、`shape_create/update/delete`、`entity_generate`、`undo/redo`、`clarify_triggered`、`asr_latency`、`parse_latency`、`render_latency`。
